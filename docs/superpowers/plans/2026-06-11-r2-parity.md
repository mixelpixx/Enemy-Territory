## Task 1: Wire CM_PointContents (and un-stale CM_DrawDebugSurface) through the bridge

The only known *code* gap. Their tr_bsp.c light-grid setup calls `ri.CM_PointContents(origin, 0)` to skip light-grid points inside solid; our bridge stub returns 0 (never solid) → subtly wrong ambient/directed light on every map. Also: the bridge no-ops `CM_DrawDebugSurface` with a comment claiming it's "not on our v9 ri" — it IS (`src/renderer/tr_public.h:199`, seeded at `cl_main.c:3235`); wire it (used by their r_showSurfaces-class debug paths) or at minimum fix the false comment.

**gl1-safety argument (engine touch):** appending one function pointer to the END of `refimport_t` and seeding it in CL_InitRef is additive-only — gl1's statically-linked renderer never reads the new field, its existing field offsets are unchanged (append-only), and no gl1 code path is edited. REF_API_VERSION bumps 9→10 so a stale DLL can't bind a mismatched struct; gl1 and the bridge compile from the same tree so both sides move together.

**Files:**
- Modify: `src/renderer/tr_public.h` (refimport_t append + REF_API_VERSION 9→10)
- Modify: `src/client/cl_main.c` (seed `ri.CM_PointContents` in CL_InitRef, next to the existing `ri.CM_DrawDebugSurface` at ~3235)
- Modify: `src/renderer2/bridge/tr2_bridge.h` (neutral-contract entry), `tr2_bridge_ours.c` (BrdgOur callback), `tr2_bridge_theirs.c` (replace the stub + the DrawDebugSurface no-op)
