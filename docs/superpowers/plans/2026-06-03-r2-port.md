# R2-2: ET:Legacy renderer2 Port Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Vendor ET:Legacy v2.84.0's renderer2 (GLSL, GL 3.3 core) into `src/renderer2/`, build it as our `etrm_renderer2.dll`, and boot ET-RM to a **fully working main menu under `cl_renderer gl2`** — with the gl1 default untouched.

**Architecture:** The vendored renderer2 compiles against **its own vendored headers** (ET:Legacy's tr_public.h/tr_types.h/q_shared subset live privately inside the DLL); ALL translation between ET-RM's engine (REF_API_VERSION 9) and renderer2's expectations (their v10 interface) happens in a **boundary shim** (`tr2_bridge.c`) — their code is never patched beyond mechanical MSVC fixes, our engine headers never change. The shim translates refimport (cvar proxies, function stubs/adapters, GLimp signature mapping + GLEW init) and refexport (glconfig copy-back, signature wraps). GLSL shaders embed via ET:Legacy's own `shdr2h` tool. This is the project's largest increment; compile-bring-up tasks are explicitly iterate-until-green with escalation criteria.

**Tech Stack:** C/C++ (shdr2h is C++), CMake + Ninja + MSVC x64, GLEW (vendored), SDL2 (engine-side only), ET:Legacy v2.84.0 @ 764ffc00a953e59aaf435272d004c49a89710309 (reference clone at `C:\repo\et-deps\etlegacy`, GPL-3.0 — same license as our base; THIRDPARTY policy: preferred verbatim port source).

## Recon facts the tasks rely on (verified against the pinned clone)

- Their build: `cmake/ETLBuildRenderer.cmake` lines ~120-171 — target = renderer2 .c files + renderercommon (minus vulkan/svg) + generated `shaders.h`; `FEATURE_RENDERER2` + `USE_REFENTITY_ANIMATIONSYSTEM=1` defines; links GLEW + jpeg/png/zlib/freetype.
- GLSL embed: host tool `src/renderer2/glsl/shdr/shaders2h.cpp` → `shdr2h LEGACY <glsl dir> <gldef> <out shaders.h>`; `tr_glsl.c` includes `<shaders.h>`.
- GL: 3.3 core (their tr_init.c builds a glConfigString with major=3 minor=3 context=CORE); GLEW resolves entry points (`glewInit()` after context creation).
- Window: THEIR engine owns it; renderer calls `ri.GLimp_Init(glconfig_t*, const char *glConfigString)` and `ri.GLimp_SwapFrame()`. Ours provides `GLimp_Init(const glimpParams_t*)` / `GLimp_EndFrame()` — shim maps (parse nothing: shim hardcodes 3.3 core request into glimpParams, retries compat via the engine ladder).
- Drift map TOP items (full list in the recon report, session history): (1) their cvar_t = 19 fields vs ours 11 + 64-bit flags — their code reads `cvar->value/integer/modified` directly → shim keeps a registry of THEIR-layout proxy cvars, syncing value/integer/string/modified from ours each frame + on Cvar_Get; (2) their glconfig_t has appended fields (GLSL version, windowWidth/Height etc.) → renderer2 writes ITS glconfig; shim copies the common prefix back into the engine's `cls.glconfig` at BeginRegistration; (3) ~11 refimport functions we lack (FS_FOpenFileRead/FS_Read, RealTime, CL_VideoRecording/WriteAVI, Sys_GLimp*/Sys_SetEnv, IN_Init/Shutdown/Restart, zlib_*, Cvar_CheckRange/SetDescription/VariableIntegerValue, Cmd_AddSystemCommand, CM_PointContents) → shim supplies adapters (FS via our FS_ReadFile where possible, else real adapters) or safe stubs (AVI=qfalse/no-op, IN_*=no-op since OUR engine owns input, Sys_*=no-op, CheckRange/SetDescription=no-op); (4) refexport drift: their BeginFrame(void) vs our BeginFrame(stereoFrame_t), their RegisterFont 4-arg, missing SaveViewParms/RestoreViewParms (stub no-ops; flag for R2-3 parity), GLimp swap naming → shim wraps every refexport member into OUR v9 shapes.
- Scene-data structs (refdef_t/refEntity_t/polyVert_t/polyBuffer_t/glfog_t) cross the boundary as POINTERS from our cgame → layout must MATCH or be translated per-call. Task 4 builds an automated layout-diff checker before deciding pass-through vs translate.
- Deps: tr_image_jpg (libjpeg — we have bundled jpeg-6b; theirs may want newer API), tr_image_png (libpng+zlib — we have NEITHER), tr_font (freetype — we have bundled old ft2). **Descope valve for boot-to-menu:** PNG support may be stubbed (retail ET assets are TGA/JPG; menu doesn't need PNG) and revisited in R2-3; freetype is needed only if their tr_font won't compile against a minimal path — investigate in Task 3, FetchContent freetype2 if required (zlib license, log it).

- [ ] **Step 2:** CVAR PROXY LAYER (the highest-risk drift item): bridge `Cvar_Get` adapter allocates a THEIR-layout cvar struct per name (registry hash), calls our Cvar_Get underneath, copies name/string/value/integer/flags(narrowed)/modified, returns the proxy. `Cvar_Set` adapters write through to ours then refresh the proxy. A per-frame sync (hooked in the BeginFrame wrap) refreshes value/integer/string/modified for all proxies so their direct field reads stay current; their writes to proxy fields (rare — `->modified = qfalse` pattern) sync back for `modified` specifically. - [ ] **Step 4:** LAYOUT CHECKER (`tr2_layoutcheck.c`): compile-time `static_assert`/`Com_Error`-at-init comparisons for every struct passed BY POINTER across the boundary uncopied (refdef_t, refEntity_t, polyVert_t, polyBuffer_t, glfog_t, fontInfo_t...): sizeof + offsetof of each field, OUR definition vs THEIRS (the bridge TU can see both via careful include isolation — two helper TUs if one TU can't).
